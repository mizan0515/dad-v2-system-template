# 07. 기존 프로젝트 도입 / 마이그레이션

## 목적

이미 운영 중인 저장소에 DAD v2를 심을 때, 기존 규칙/문서/자동화와 충돌하지 않게 도입 순서를 정리한다.

## 기본 점검 항목

1. 기존 루트 규칙 문서와 `PROJECT-RULES.md`, `AGENTS.md`, `CLAUDE.md`가 충돌하지 않는지
2. 기존 slash command, agent skill, prompt, automation 체계가 이미 있는지
3. `Document/` 또는 유사 폴더에 세션/리뷰/운영 문서가 이미 있는지
4. git 정책, 브랜치 정책, PR 정책이 템플릿 기본값과 다른지
5. 저장소 전용 research / inventory / architecture 문서가 어디에 있는지
6. 기존 validator, linter, test runner와 DAD validator를 어떻게 함께 돌릴지

## 도입 절차

1. authoritative 문서를 먼저 정한다. 기존 문서가 있으면 무엇을 승계하고 무엇을 대체할지 명시한다.
2. `PROJECT-RULES.md`를 해당 저장소의 실제 규칙으로 채운다.
3. `AGENTS.md`, `CLAUDE.md`, `DIALOGUE-PROTOCOL.md`를 저장소 환경에 맞게 조정한다.
4. `.claude/commands/`, `.agents/skills/`, `.prompts/`, `Document/` 운영 가이드를 루트 계약과 맞춘다.
5. 저장소 전용 Codex skill namespace를 적용하고, `.agents/skills/` 메타데이터를 검증한 뒤 Codex Desktop 스킬을 등록한다 (`Set-CodexSkillNamespace` 후 `Validate-CodexSkillMetadata`, 그 다음 `Register-CodexSkills`).
6. `tools/Validate-Documents.ps1 -Root . -IncludeRootGuides -IncludeAgentDocs -Fix`를 먼저 통과시킨다.
7. 필요하면 샘플 세션 하나로 `New-DadSession`, `New-DadTurn`, `Validate-DadPacket` 스모크 검증을 한다.
8. live 작업 전 시스템 감사 프롬프트(`01`)로 전체 정합성을 다시 점검한다.

## 병합 전략

기존 저장소에 이미 `CLAUDE.md`, `AGENTS.md`, `.claude/commands/` 등이 있을 때:

1. 기존 파일을 백업한다 (`{파일명}.pre-dad.bak`).
2. DAD v2 섹션을 기존 파일에 **추가(append)** 하되, 기존 규칙을 삭제하지 않는다.
3. 충돌 지점(예: git 정책, 브랜치 규칙, 검증 절차)을 명시적으로 나열하고 어느 쪽을 따를지 결정한다.
4. 결정 결과를 `PROJECT-RULES.md`에 기록한다 (어떤 규칙을 승계했고, 어떤 규칙을 대체했는지).
5. 병합 후 `tools/Validate-Documents.ps1 -Root . -IncludeRootGuides -IncludeAgentDocs -Fix`를 통과시킨다.

## 롤백 경로

병합 후 validator나 운영 스모크가 깨지면:

1. `.pre-dad.bak`로 백업한 원본 파일을 즉시 복원한다.
2. DAD 관련 신규 디렉터리(`.claude/commands/`, `.agents/skills/`, `.prompts/`, `Document/dialogue/`, `tools/`) 중 이번 도입에서 추가한 항목만 제거하거나 되돌린다.
3. 어떤 파일을 복원했고 어떤 DAD 항목을 제거했는지 기록한다.
4. 충돌 원인을 `PROJECT-RULES.md` 또는 별도 운영 메모에 남긴 뒤, 다음 도입 시 병합 전략을 수정한다.

## 금지

- 기존 프로젝트 규칙을 읽지 않고 템플릿 문서로 덮어쓰기
- 저장소 전용 검증 절차를 무시한 채 DAD validator만 통과시키기
- 명령/스킬/프롬프트/validator 중 일부만 옮겨 놓고 시스템 도입이 끝났다고 간주하기
