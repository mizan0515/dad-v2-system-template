# 02. 세션 시작 / 컨트랙트 작성

## 목적

새 DAD v2 세션을 시작할 때 under-scope 없이, 그리고 meta-only work로 흐르지 않게 첫 Turn의 Contract와 실행 범위를 잡는다.

## 언제 사용하나

- 새 세션을 시작할 때
- 기존 세션을 supersede하고 새 세션으로 갈아탈 때
- 사용자의 요청이 짧거나 모호해서 곧바로 체크포인트로 내리기 어려울 때

## 수행 순서

1. `PROJECT-RULES.md`, `AGENTS.md` 또는 `CLAUDE.md`, `DIALOGUE-PROTOCOL.md`를 읽는다.
2. 라이브 파일 상태와 현재 브랜치를 확인한다.
3. 제안된 세션이 outcome-scoped인지 먼저 판별한다. wording correction, summary/state sync, closure seal, validator-noise cleanup만의 작업이라면 broken DAD state를 명시적으로 복구하는 경우가 아닌 한 현재 execution session 안에 흡수한다.
4. `Document/dialogue/backlog.json`이 있으면 함께 확인한다. 새 세션은 backlog item 하나에 연결하거나, 완전히 새로운 작업이면 기존 `now` item이 없을 때만 auto-bootstrap한다. 다른 active session이 아직 열려 있으면 새 candidate는 대기시키고 즉시 승격하지 않는다.
5. 필요하면 `task_model`을 먼저 만든다.
6. `success_shape`, `major_risks`, `out_of_scope`를 기준으로 3~5개 체크포인트를 만든다.
7. Turn 1에서 가능한 첫 수직 슬라이스까지 같이 실행한다.
8. Turn Packet에 Contract 초안과 첫 실행 근거를 함께 남긴다.
9. 다음 peer 턴이 남아 있으면 `handoff.closeout_kind: peer_handoff`를 설정하고 relay prompt artifact를 저장한 뒤, 같은 프롬프트를 그 턴을 닫는 같은 최종 응답에 그대로 붙인다. 사용자가 다음 프롬프트를 다시 요청하게 만들지 않는다.

## 체크포인트 품질 기준

- 구현 산출물 기준으로 판정 가능해야 한다.
- 최소 1개 checkpoint는 왜 이 세션을 열었는지 설명하는 concrete outcome 또는 artifact를 직접 가리켜야 한다.
- 검증 방법이 checkpoint에 같이 들어가야 한다.
- `out_of_scope`를 몰래 다시 끌어오면 안 된다.
- 프로젝트 문서 drift가 보이면 별도 checkpoint 또는 후속 작업으로 드러내야 한다.
